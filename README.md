# 基于《千恋＊万花》角色设定的 AI 小镇恋语 RPG 原型项目

> 一个以 **Unreal Engine 5.7** 为核心、围绕 **AI 驱动 NPC 交互 + 小镇日常模拟 + 战斗成长闭环** 构建的二创原型项目。
> 本项目重点展示了我在 **游戏客户端架构设计、核心系统开发、LLM 交互链路接入、状态持久化与多模块联调** 方面的完整实践能力。

---

## 项目简介

本项目以《千恋＊万花》的角色设定为灵感，构建了一个可运行的 **AI 小镇 RPG 原型**。玩家可以在小镇中探索、与特殊 NPC 对话、提升关系、参与战斗、获取掉落、管理背包与商店，并通过存档系统保持关键状态一致性。

项目的目标并不是简单复刻原作内容，而是围绕“**如何将大模型能力自然地嵌入 RPG 交互体验**”这一方向，完成一个具备较完整客户端系统的原型验证。

---

## 游戏背景与世界观

项目以《千恋＊万花》的角色设定为灵感来源，将原作中具有鲜明个性的角色放入一个可自由探索的小镇RPG原型中，尝试回答这样一个问题：

**如果拥有稳定角色设定的大模型 NPC 真的生活在一个会随时间变化、存在战斗与资源循环的小镇里，玩家会得到怎样的互动体验？**

在这个原型中，小镇既是日常生活空间，也是系统联动的核心容器：

* 白天与夜晚会影响天空、环境氛围与NPC的行为逻辑
* 特殊 NPC 拥有独立人设、关系状态与对话表现
* 玩家既可以进行日常交流、关系培养，也可以进入战斗、完成任务、获取资源
* 角色关系、战斗成长、物品获取与世界状态并不是割裂存在，而是围绕同一套客户端系统持续联动

项目整体风格定位为：
**日式小镇氛围 + 角色关系交互 + 轻量战斗成长 + AI 驱动对话体验**。

---

## 核心玩法与体验流程

当前原型的主要玩法循环可以概括为：

**探索小镇 → 与 NPC 互动 → 提升关系 → 触发属性反馈 → 参与战斗 → 获取掉落 → 管理背包/商店/任务 → 推进时间与世界状态**

围绕这条主循环，项目重点验证了以下几类体验：

### 1. 小镇日常体验

玩家可以在小镇中自由移动，与特殊 NPC 进行对话，感受不同角色在时间、关系与情境影响下的回应差异。这里的重点并不是传统固定文本对话，而是让 NPC 在保持角色设定的前提下，具备更自然的表达能力与关系反馈。

### 2. 关系养成体验

特殊 NPC 的关系变化不会只停留在文本层面，而是会进一步作用到角色成长与系统状态。例如角色关系提升后，会对主角的部分属性产生联动反馈，使“交流”真正进入玩法闭环，而不是仅作为剧情装饰。

### 3. 战斗与成长体验

玩家可以在小镇外或特定区域与敌人战斗，击败敌人后获得掉落与任务推进。战斗系统与角色属性、背包道具、任务目标相互关联，形成基础成长反馈。

### 4. 资源循环体验

通过战斗获取资源，再进入背包、使用道具、前往商店购买物品，构成一条轻量但完整的资源循环。该部分用于验证客户端中战斗、掉落、物品、商店、UI 与存档的系统整合能力。

### 5. 世界状态推进体验

游戏时间会持续推进，并影响天空状态、NPC行为、怪物刷新与部分系统表现；通过保存、读档、时间跳转等场景，进一步验证世界状态是否能够保持一致。

## 玩法与系统设计补充

除了客户端功能实现外，本项目也强调“玩法表达是否成立”。从设计上看，项目并非把多个系统简单拼接在一起，而是尝试建立一个以小镇为核心舞台的轻量 RPG 闭环：

* **角色交互** 提供情绪反馈与关系变化
* **关系变化** 影响属性与成长反馈
* **战斗与任务** 提供目标驱动与资源来源
* **掉落、背包、商店** 提供资源管理与循环承接
* **时间与世界状态** 让静态场景具备持续运行感

这种设计使项目既能展示客户端系统实现能力，也能体现本人对游戏体验结构与模块协同关系的理解。

## 关键技术设计

### 一、Gameplay 架构

项目基于 UE 常规 Gameplay 体系组织逻辑：

* **GameMode**：规则入口与局部流程管理
* **GameState**：时间、全局状态与部分广播
* **PlayerController**：UI、输入模式与菜单管理
* **Character**：角色状态、任务进度、交互桥接
* **Actor / Component**：敌人、掉落物、背包、商店等功能模块拆分

这种设计方式使得后续系统扩展时，能够较清晰地划分“输入层、表现层、状态层、规则层”的职责。

---

### 二、LLM 交互链路

我在项目中将大模型能力封装为一条可控的交互链路，而不是简单调用聊天接口。

主要包括：

* 角色 Prompt 模板
* 角色条件信息注入
* 对话历史管理
* 响应清洗与解析
* 数值反馈映射

这样做的目的是让模型输出不仅“像在说话”，还能够服务于游戏系统本身。

---

### 三、状态持久化设计

项目在后期的重点之一，是保证多模块并行开发后仍能实现稳定读档。

当前存档系统已覆盖：

* 时间
* 战斗属性
* 背包与金钱
* 任务进度
* NPC 好感度
* 商店库存

相比只做位置与时间恢复，这一部分更偏向客户端系统整合与工程稳定性验证。

## 本地模型运行与性能表现

项目当前支持在本地部署模型环境下运行，并兼顾了原型阶段对“可用性”与“响应速度”的需求。

* 在 **16GB 显存显卡** 环境下，可稳定运行游戏客户端，并同时加载 **Qwen2.5-7B** 模型用于 NPC 对话
* 当前优先采用 **非深度思考类模型** 作为交互模型，以提升对话响应速度和实际可玩性
* 在该配置下，模型推理速度可达到约 **30 token/s**，NPC 单次回复的平均等待时间约为 **1 秒**
* 若切换至约 **1.5B 参数量** 的轻量模型（且模型不额外加载至内存），生成速度可达到 **100+ token/s**，NPC 单次回复几乎无需等待，但是对话质量有一定的下滑，内容丰富性一般

这一部分验证了本项目在个人开发环境下，将本地大模型直接接入游戏原型并实现较流畅实时交互的可行性，也体现了在模型规模、响应速度与玩家体验之间进行工程权衡的思路。
